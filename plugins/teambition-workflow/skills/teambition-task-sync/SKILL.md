# Teambition 任务评论/完成

## 前提条件

* 已配置 `teambition-mcp` MCP 服务器
* 代码修改已完成并提交
* 当前会话可访问目标代码仓库
* 可读取目标 Teambition 任务详情、项目、自定义字段和状态

## 模式判断

| 用户意图 | 模式 | 操作范围 |
|----------|------|----------|
| "评论任务"、"记录一下"、"备注"、"tb评论"、"teambition评论" | 仅评论 | 仅添加评论 |
| "完成任务"、"解决bug"、"已解决"、"已完成"、"tb完成"、"teambition完成" | 完整完成 | 更新字段 + 修改状态 + 添加评论 |
| 无法判断 | 询问用户 | — |

### 触发约束

以下情况不要直接执行，而是先询问用户：

* 只说"已完成""已解决""bug 修好了"
* 只有状态讨论，没有明确要求执行 Teambition 操作
* 没有任务链接、任务 ID，且没有明确 Teambition/TB 指代

## 配置

### 工作目录与项目映射

按以下优先级读取：

1. **CLAUDE.md**：全局 `~/.claude/CLAUDE.md` 或项目级 `CLAUDE.md` 中的项目列表
2. **插件本地**：`${CLAUDE_PLUGIN_ROOT}/references/projects.md`
3. **默认**：`~/workspace`，无项目映射

**CLAUDE.md 配置示例**：

```markdown
## 工作区项目

`~/workspace` 下的主要项目：

| 项目 | 说明 |
|-----|------|
| backend-api | 后端 API 服务 |
| admin-web | 管理后台前端 |
```

**项目定位优先级**：用户指定 > 当前工作目录 > MCP 获取任务所属项目名后匹配映射表 > 询问用户

### 署名

所有评论和字段更新的内容末尾添加署名：`*Powered by USERNAME's AI Assistant*`

USERNAME 来源（优先级从高到低）：

1. CLAUDE.md 中的配置
2. `git config user.name`
3. 若均无则使用 `AI Assistant`

## 执行步骤

### Step 0: 前置检查

在任何写入动作前，先完成以下检查：

1. **MCP 可用性检查**
   * 能读取任务详情、项目信息、自定义字段、当前状态
   * 能创建任务评论
   * 完整完成模式下还能修改字段和状态

2. **任务有效性检查**
   * 任务存在且可访问
   * 可读取任务标题、所属项目、当前状态
   * 可读取任务类型或足以判断类型的信息

3. **Git 仓库检查**
   * 目标目录存在
   * 目标目录是有效 Git 仓库
   * 可读取当前分支、提交记录和 diff

若任一关键检查失败：
* 停止字段更新和状态更新
* 尽量降级为仅生成评论草稿
* 明确告知用户失败原因

### Step 1: 解析输入

从用户输入中提取任务信息：

| 参数 | 必填 | 说明 | 示例 |
|------|------|------|------|
| 任务链接/ID | 是 | Teambition 任务 URL 或 ID | `https://www.teambition.com/task/12345` |
| 项目目录 | 否 | 工作目录下的项目目录名 | `my-project` |
| 分支名 | 否 | Git 分支名称 | `release/v1.0` |
| 模式 | 否 | 仅评论 / 完整完成 | `tb评论` |
| 部署说明 | 否 | 用户额外提供的部署注意事项 | `需先执行数据脚本` |

示例：

```
完成任务 https://www.teambition.com/task/12345
评论任务 https://www.teambition.com/task/12345 my-project release/v1.0
解决bug https://www.teambition.com/task/67890
```

### Step 2: 获取 Git 信息

#### 2.1 分支定位

按以下优先级确定分支：

1. 用户指定分支
2. 当前仓库当前分支
3. 若任务字段中存在开发分支字段，可作为参考，但不覆盖用户明确输入

#### 2.2 基线分支定位

按以下优先级尝试识别基线分支：

1. `origin/main`
2. `origin/master`
3. `origin/develop`
4. 项目默认主干分支
5. 用户明确指定的基线分支

#### 2.3 推荐命令

**用户只提供任务链接**（未指定项目和分支）：

```bash
git branch --show-current
git fetch --all
git merge-base HEAD origin/main
git log --oneline <merge-base>..HEAD
git diff --stat <merge-base>..HEAD
```

**用户指定项目和分支**：

```bash
cd <工作目录>/<项目目录>
git fetch --all
git merge-base <分支名> origin/main
git log --oneline <merge-base>..<分支名>
git diff --stat <merge-base>..<分支名>
```

**需要深入理解功能时**：

```bash
git show <branch> --stat --no-patch
git diff <merge-base>..<branch> | head -200
```

#### 2.4 提取规则

* 只提取分支相对基线分支独有的提交，不包含从其他分支合并进来的改动
* 分支名含日期时（如 `release/v20260402-xxx`），通常只关注最新 1-3 个与业务有关的提交，但仍以 diff 为准
* 分析代码改动后提炼摘要，不是直接评论分支名
* 若无法可靠判断用户可感知变化，使用保守表述，不编造功能点

### Step 3: 生成改动摘要

**核心原则**：从业务角度描述，让产品经理和测试人员能看懂。

| ✅ 推荐 | ❌ 避免 |
|---------|---------|
| 用户列表页新增「状态筛选」功能，支持按启用/禁用筛选 | 修改了 UserController，新增 statusFilter 属性 |
| 详情页新增「操作日志」标签页，记录所有状态变更历史 | 更新了相关视图文件 |
| 新增「负责人」字段，必填 | 添加了 form 字段 |
| 修复了订单超时后仍可重复提交的问题 | 改了后端接口逻辑 |

生成规则：

* 优先使用 commit message 中的描述，描述功能变化而非文件修改
* 优先描述：新增能力、修复问题、调整规则、补充校验、部署注意事项
* 避免描述：类名、文件名、组件名、函数名等技术细节
* 不要写未经验证的"已全面修复""已验证通过""可以直接上线"
* 不确定时使用保守表达：调整了……、补充了……、优化了……、修复了某场景下的问题……

### Step 4: 执行操作（按模式）

#### 仅评论模式

跳过字段和状态更新，直接到 Step 5 添加评论。

#### 完整完成模式

##### 4.1 更新自定义字段

通过 MCP 工具查询项目自定义字段（如 `SearchProjectCustomFieldsV3`），有则更新。

| 任务类型 | 常见字段名 | 填写内容 |
|----------|-----------|----------|
| 缺陷/Bug | 解决方案/修复说明/处理结果/修复内容 | 修复内容摘要 + 署名 |
| 缺陷/Bug | 问题原因/根因/原因分析/问题分析 | 根因分析 + 署名 |
| 一般任务 | 分支/开发分支/提测分支 | 当前开发分支名称 |
| 一般任务 | 调整说明/改动说明/实现说明 | 主要调整内容 + 署名 |
| 一般任务 | 部署说明/上线说明/注意事项 | 部署注意事项（如有）+ 署名 |

说明：

* 不同项目的自定义字段名称可能不同，以实际查询结果为准
* 无对应字段则跳过
* 按字段语义匹配，不要求字段名完全一致
* 不要为了填满字段而编造内容

##### 4.2 更新任务状态

先判断任务类型，优先级如下：

1. 任务类型字段
2. 工作流/任务列表归属
3. 标签
4. 标题关键词（仅作兜底）

状态更新规则：

* Bug 任务 → `已解决`
* 一般任务 → `已完成`
* 若当前已是目标状态，不重复更新
* 若无法可靠判断任务类型，不自动修改状态，可继续仅评论

> **硬约束**：Bug 任务必须先成功更新"问题原因"和"解决方案"字段后，才能改为"已解决"。字段更新失败则中止，不更新状态。

### Step 5: 添加评论

通过 MCP 工具（如 `CreateTaskCommentV3`）添加评论：

```markdown
**分支**: [分支名]

**改动摘要**:
- [功能点1]
- [功能点2]

**部署说明**: [如有则填写，无则省略]

---
*Powered by USERNAME's AI Assistant*
```

评论要求：

* 分支名必须明确
* 改动摘要控制在 2-5 条
* 摘要需对业务方可理解
* 没有部署要求时，省略"部署说明"整段
* 不写未经验证的承诺性表述

### Step 6: 输出总结

```markdown
## 任务完成

**任务标题**: [标题]
**操作模式**: [仅评论/完整完成]
**任务状态**: [保持不变/已解决/已完成/更新失败]
**修改分支**: [分支名]
**主要改动**: [改动摘要]
**已添加评论**: ✓ / ✗
**字段更新**: [已更新/部分更新/未更新]
**状态更新**: [成功/跳过/失败]
**说明**: [补充说明]
```

## 失败回退

遵循"尽量保留成果、尽量少做危险写入"的原则。

1. **字段更新失败**
   * Bug 任务：不要更新状态
   * 一般任务：可继续尝试仅评论，并说明字段未更新

2. **状态更新失败**
   * 若评论尚未写入，可继续尝试写评论
   * 输出中明确说明"评论已写入，状态未更新"

3. **评论写入失败**
   * 不要报告"任务已同步完成"
   * 输出评论草稿供用户复制
   * 若状态已经更新，必须明确提示"状态已更新但评论失败"

4. **Git 信息不足**
   * 不要编造改动摘要
   * 可降级为简短评论草稿，例如"已处理，待补充分支和改动说明"
   * 只有在用户明确允许时才实际发布这类弱信息评论

5. **项目目录或分支无法定位**
   * 停止自动写入
   * 返回需要用户补充的信息

## 幂等与重复保护

执行前检查以下情况：

* 任务是否已经处于目标状态
* 最近是否已有相似评论
* 相同分支是否已经同步过

若检测到高度重复：

* 默认不重复写入
* 向用户说明已存在类似同步记录
* 仅在用户明确要求时再次评论

## 行为约束

* 不要因为泛化表达单独触发
* 不要在任务、项目、分支三者都不清楚时直接写入外部系统
* 不要在没有可靠依据时自动把任务判断为 Bug
* 不要在 Bug 缺少根因和解决方案字段时改为"已解决"
* 不要编造业务功能、验证结论或上线结论
* 不要仅凭最近几条 Git 提交就判断任务已完成
* 不要在目录匹配不唯一时自动选择仓库
* 不要把纯技术改动直接翻译成业务价值结论

## 常见示例

### 示例 1：仅评论

输入：

```
评论任务 https://www.teambition.com/task/12345
```

行为：

* 读取任务
* 自动定位当前仓库和当前分支
* 读取 Git 独有改动
* 生成业务可读评论
* 添加评论
* 不更新字段，不改状态

### 示例 2：指定项目和分支评论

输入：

```
tb评论 https://www.teambition.com/task/12345 admin-web release/v1.0
```

行为：

* 在指定项目目录中读取指定分支
* 提取该分支相对基线的独有改动
* 添加评论
* 不更新字段，不改状态

### 示例 3：完整完成普通任务

输入：

```
完成任务 https://www.teambition.com/task/12345
```

行为：

* 读取任务、自定义字段、当前状态
* 自动定位项目和分支
* 生成改动摘要
* 更新可匹配字段
* 将状态改为 `已完成`
* 添加评论

### 示例 4：完整完成 Bug 任务

输入：

```
解决bug https://www.teambition.com/task/67890
```

行为：

* 判断是否为 Bug 任务
* 生成根因与解决方案摘要
* 成功写入"问题原因"和"解决方案"字段后
* 将状态改为 `已解决`
* 添加评论
