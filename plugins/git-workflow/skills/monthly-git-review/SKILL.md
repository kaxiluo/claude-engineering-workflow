# 基于 Git 的月度工作总结生成

## 目标

基于指定月份内的 Git 提交记录，生成一份中文月度工作总结草稿，突出研发产出、问题修复、优化改进和协作支持，避免直接堆砌 commit message。

该技能生成的是**基于代码提交的月度工作总结草稿**，不是完整绩效事实。输出后应提醒用户按需补充非代码工作，如方案设计、需求评审、联调支持、线上排障、跨团队沟通、带教等内容。

## 配置

按以下优先级读取工作目录和项目映射：

1. **CLAUDE.md**：全局 `~/.claude/CLAUDE.md` 或项目级 `CLAUDE.md` 中的项目列表
2. **插件本地**：`${CLAUDE_PLUGIN_ROOT}/references/projects.md`
3. **当前工作目录**
4. **默认**：`~/workspace`，扫描主要项目目录，不盲目纳入所有历史仓库

**CLAUDE.md 配置示例**：

```markdown
## 工作区项目

`~/workspace` 下的主要项目：

| 项目 | 说明 |
|-----|------|
| backend-api | 后端 API 服务 |
| admin-web | 管理后台前端 |
```

## 适用范围

适合以下请求：

* 基于 Git 生成本月工作总结
* 生成上月研发月报
* 按提交记录整理月度产出
* 基于代码记录汇总本月完成内容
* 生成技术月报草稿

不适合以下场景直接执行：

* 纯模板写作，不需要读取 Git
* 正式绩效自评、晋升述职、360 反馈等非月报场景
* 非研发岗位总结
* 用户只是在问"月报怎么写"而不是要生成内容

## 执行步骤

### Step 1: 确定月份

按以下优先级确定统计范围：

1. 用户明确指定月份，直接使用
2. 用户说"本月"，使用当前自然月
3. 用户说"上月"，使用上一个完整自然月
4. 用户未指定时，默认使用**上一个完整自然月**

输出中要明确写出统计月份，例如：`2026年3月月度工作总结`

### Step 2: 确定 Git 身份

按以下优先级识别 Git 作者身份：

1. `CLAUDE.md` 中显式配置的姓名或邮箱
2. 当前仓库的 `git config user.name`
3. 当前仓库的 `git config user.email`

若用户名和邮箱都可获取，优先同时作为候选，避免漏掉提交记录。

### Step 3: 确定扫描项目范围

按以下优先级确定要扫描的仓库：

1. 用户明确指定的项目
2. `CLAUDE.md` 或 `references/projects.md` 中配置的主要项目
3. 当前工作目录所在仓库
4. `~/workspace` 下近期仍活跃的主要 Git 仓库

规则：

* 优先扫描配置中的主要项目，不要默认把所有含 `.git` 的目录都纳入
* 排除 `references/projects.md` 中「排除项目」列表里的项目
* 忽略明显无关的 demo、临时仓库、历史归档仓库
* 若某仓库在目标月份没有本人提交，可跳过
* 若最终没有任何有效仓库，提示用户检查月份、作者身份或项目范围

### Step 4: 获取提交记录

对每个目标仓库执行：

```bash
git log --author="<username-or-email>" --since="<month>-01 00:00:00" --until="<month>-last-day 23:59:59" --oneline
```

当结果过少或疑似遗漏时，可追加尝试：

```bash
git log --author="<user.email>" --since="<month>-01 00:00:00" --until="<month>-last-day 23:59:59" --oneline
```

必要时对重点提交进一步查看：

```bash
git show --stat --no-patch <commit>
git show <commit> | head -200
```

### Step 5: 过滤和归类提交

先过滤低价值提交，再归类有效工作内容。

#### 5.1 过滤规则

默认忽略以下提交，除非它们明显对应重要工作：

* merge
* typo
* fix typo
* format
* lint
* prettier
* style only
* update readme
* bump version
* chore
* 单纯格式调整
* 无业务含义的机械修改

#### 5.2 归类规则

按项目分组后，在每个项目内按主题归类。优先依据以下信号综合判断：

1. commit message 中的功能或问题关键词
2. 分支名中的需求、模块、版本信息
3. 修改路径中的模块目录
4. 相邻提交的主题相似性

优先归为以下稳定类别之一：

* 功能开发
* 问题修复
* 流程或体验优化
* 基础能力或重构
* 维护与支持

若能够明确识别业务模块，也可在类别下细分为更具体的主题。

#### 5.3 合并规则

* 同一项目内，同一主题的多个提交合并为一条成果
* 多个零碎修复可汇总为一个"问题修复/稳定性优化"主题
* 不要逐条罗列 commit
* 不要机械复制提交信息

### Step 6: 提炼工作总结

将提交记录转写成业务可读、面向团队或主管可理解的总结内容。

#### 核心原则

* 用中文输出
* 用业务语言描述
* 强调完成了什么、解决了什么、带来了什么效果
* 不直接罗列 commit message
* 不堆砌技术名词、类名、文件名
* 不夸大影响，不编造没有证据的价值

#### 推荐表达

* 完成了后台用户管理模块的状态筛选和详情能力补充，提升了运营排查效率
* 修复了订单处理中的异常场景，减少重复提交和状态不一致问题
* 优化了提交流程中的必填校验和交互提示，降低了错误数据进入流程的概率
* 补充了部分基础能力和兼容性处理，提升了模块稳定性

#### 避免表达

* 修改了 controller
* 更新了 service
* 调整了若干文件
* 修复了一些 bug
* 做了一些优化

当无法明确推出业务价值时，使用保守表达：

* 调整了某模块的处理逻辑
* 补充了关键校验与异常处理
* 优化了部分流程的稳定性和可用性
* 修复了若干已知问题

### Step 7: 生成输出

默认生成"简洁月报型"工作总结，结构如下：

```markdown
2026年3月月度工作总结

一、本月主要工作

1. [项目或主题]
[用业务语言概括完成内容、解决的问题和实际效果]

2. [项目或主题]
[概括主要产出]

二、问题修复与优化

- [归纳本月处理的问题或优化事项]
- [归纳稳定性、体验或流程层面的改进]

三、总结

本月主要围绕[核心方向]开展工作，完成了[核心成果概括]，同时处理了[问题修复/优化事项]，整体推动了[业务或系统效果]。
```

当项目较多时，可按"项目维度"输出；当项目较少但主题清晰时，可按"主题维度"输出。

## 规则

* 所有输出使用中文
* 默认输出的是**月度工作总结草稿**
* 不直接罗列 commit message
* 不逐条列出提交 hash
* 不把提交数量等同于工作价值
* 若某月没有提交记录，提示用户检查月份、Git 身份或项目范围
* 若提交记录较少，也可以输出简短总结，但需明确说明"本月代码提交较少，建议补充非代码工作"
* 输出后提醒用户补充非代码工作内容，如方案设计、评审协作、联调支持、线上排障、沟通推进、知识沉淀等

## 失败回退

### 1. 月份未识别

* 优先按"上一个完整自然月"处理
* 若用户表述明显歧义，再询问确认

### 2. Git 身份不明确

* 依次尝试 `user.name` 和 `user.email`
* 若仍无结果，提示用户检查 Git 配置或提供作者信息

### 3. 仓库过多或结果噪音大

* 优先仅使用配置中的主要项目
* 忽略无关仓库
* 必要时提示用户指定项目范围

### 4. 提交信息过于简略

* 对重点提交抽样查看 `git show --stat --no-patch`
* 仍无法判断时，使用保守总结，不编造功能结论

### 5. 没有有效提交

明确提示：

* 该月份未检索到有效 Git 提交记录
* 请检查月份是否正确
* 请检查 Git 用户名或邮箱是否匹配
* 请确认目标项目是否包含在扫描范围内

## 常见示例

### 示例 1：生成上月月报

输入：

```
帮我基于 git 生成上月工作总结
```

行为：

* 默认识别"上月"为上一个完整自然月
* 扫描主要项目中的本人提交
* 归类并生成中文月度工作总结草稿

### 示例 2：生成指定月份总结

输入：

```
生成 2026-03 的月度工作总结
```

行为：

* 使用 2026 年 3 月作为统计范围
* 汇总当月本人在各项目中的提交
* 按项目和主题提炼总结

### 示例 3：指定项目总结

输入：

```
基于 backend-api 和 admin-web 的提交整理本月月报
```

行为：

* 只扫描指定项目
* 汇总本月提交
* 输出聚焦这两个项目的月度工作总结

## 行为约束

* 不要因为"绩效""工作总结""月报"这类泛化表述单独触发
* 不要把该技能当作正式绩效评估工具
* 不要把 Git 提交记录包装成完整工作事实
* 不要编造业务成果或夸大影响
* 不要逐条堆砌提交记录
* 不要将低价值机械提交当作主要产出
* 不要默认扫描所有历史 Git 仓库
* 输出应可读、可改、可直接作为月报草稿使用
